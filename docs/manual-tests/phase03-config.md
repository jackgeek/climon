# Phase 3 ‚Äî `climon-config` crate

These cases prove that the ported Rust config subsystem (`climon-config`) stays
byte-for-byte compatible with the unchanged Bun/TypeScript client: it reads and
cascades config across project-local `.climon/config.jsonc` files up to the global
`$CLIMON_HOME/config.jsonc`, migrates legacy `config.json` on write (leaving a
`.bak`), writes JSONC with regenerated per-setting comments, and emits the exact
same `docs:config` section, settings table, and settings help as the Bun
generator.

Background: Phase 3 adds the `rust/climon-config` crate porting `src/config.ts`,
`src/config-jsonc.ts`, `src/config-settings.ts`, `src/features.ts`, and
`scripts/generate-config-docs.ts`. Cross-language parity is the core invariant,
enforced by the shared golden corpus under `fixtures/config/` (asserted by both
`rust/climon-config/tests/fixtures.rs` and `tests/config-fixtures.test.ts`) and
regenerated by `bun scripts/gen-config-fixtures.ts`. See the
[master plan](../superpowers/specs/2026-06-17-rust-client-rewrite-master-plan.md)
and the
[Phase 3 plan](../superpowers/plans/2026-06-19-phase03-climon-config.md).

The CLI surface (`climon config ‚Ä¶` argument parsing / command dispatch) is **not**
part of this phase ‚Äî it lands with the client CLI in a later phase. These cases
exercise the config *library* behaviour directly via the Rust + Bun test suites.

No configuration matrix applies to this phase: it is single-environment per OS.
Run the cases independently on each platform listed.

  (`rustup show` lists a `stable` default with `rustfmt` + `clippy` components).
- **Config-matrix cell:** n/a

**Steps:**
1. From the repo root: `cd rust`.
2. Build the whole workspace: `cargo build --workspace`.
3. Run the config crate tests: `cargo test -p climon-config`.

   `cargo fmt --all --check` and

  and `climon-rs`) with no errors.

  `jsonc`, `features`, and `docs` unit tests plus the `fixtures` integration
  test all pass).
- `fmt --check` reports no diffs and `clippy -D warnings` produces no warnings.

## MT-P3-02 ‚Äî Config wire parity (Bun ‚áÑ Rust golden corpus)

- **ID:** MT-P3-02
- **Feature / phase:** Phase 3 ‚Äî JSONC parse/render + settings registry
- **Preconditions:** Repo checked out; `bun install` done; `cd rust` builds (see
  MT-P3-01).
- **Config-matrix cell:** n/a
- **Platforms:** macOS, Linux, Windows

**Steps:**
1. Assert the Rust side matches the committed corpus:
   `cd rust && cargo test -p climon-config --test fixtures`.
2. Assert the Bun side matches the *same* corpus:
   `bun test tests/config-fixtures.test.ts` (from the repo root).
3. Confirm both read the identical files by inspecting `fixtures/config/`
   (`parse-cases.json`, `parse-error-cases.json`, `render-cases.json`,
   `default-config.json`, `default-rendered.jsonc`, `settings-table.md`,
   `settings-help.txt`, `docs-section.md`).
4. Regeneration check: edit one setting's `purpose`/`default` in the registry,
   run `bun scripts/gen-config-fixtures.ts`, and re-run steps 1‚Äì2. Both sides
   must update to the **same** new output and stay green. Revert the edit
   afterwards.

**Expected result:**
- Both the Rust `fixtures` integration test and `tests/config-fixtures.test.ts`
  pass against the single `fixtures/config/` corpus.
- The two implementations produce byte-identical JSONC renders (including the
  regenerated `// <purpose>` comments, registry key order, and unknown keys
  sorted alphabetically after known keys), and byte-identical settings table,
  settings help, and `docs:config` section text.

## MT-P3-03 ‚Äî Cascade resolution + legacy `config.json` migration

- **ID:** MT-P3-03
- **Feature / phase:** Phase 3 ‚Äî cascade + write target + migration
- **Preconditions:** `cd rust && cargo build --workspace` succeeds. Use a
  scratch `$CLIMON_HOME` and a scratch project tree so you never touch your real
  `~/.climon`.
- **Config-matrix cell:** n/a
- **Platforms:** macOS, Linux, Windows

**Steps:**
1. Run the cascade/load/save unit tests directly:
   `cd rust && cargo test -p climon-config config`.
2. Inspect the covered behaviours in `rust/climon-config/src/config.rs` tests:
   project-local `.climon/config.jsonc` overrides ancestor dirs which override
   the global `$CLIMON_HOME/config.jsonc`; the cwd-upward walk stops at the home
   boundary; `--local`/`--global`/auto write-target selection; sparse writes
   that preserve sibling keys and drop emptied sections.
3. Confirm legacy migration: the `save_config`/`write_config_setting` tests show
   a pre-existing `config.json` being read, then on first write a `config.jsonc`
   is produced and the legacy file is renamed to `config.json.bak`.
4. Cross-check against the Bun behaviour by skimming `tests/config-cascade.test.ts`,
   `tests/config.test.ts`, and `tests/config-global-read.test.ts` ‚Äî the Rust
   tests mirror the same scenarios.

**Expected result:**
- `cargo test -p climon-config config` is green.
- Resolution order, the home-boundary stop, write-target selection, sparse-write
  preservation, and the read-legacy ‚Üí write-`config.jsonc` + `.bak` migration all
  match the documented TypeScript behaviour.

  (`cargo install --locked cargo-deny cargo-about`). `cd rust`.
- **Config-matrix cell:** n/a

**Steps:**
1. Baseline: `cargo deny check` ‚Äî all of advisories/bans/licenses/sources are ok.

   environment restricts it):
   `cargo about generate about.hbs > NOTICES.gen.md && diff -u THIRD-PARTY-LICENSES.md NOTICES.gen.md && rm NOTICES.gen.md`

- Step 1 passes with the current dependency tree (no new licenses introduced).
- Step 2 produces no diff: the committed `THIRD-PARTY-LICENSES.md` is exactly
  what `cargo about` regenerates (idempotent).
- `climon-config` is attributed and the license summary count is unchanged apart
